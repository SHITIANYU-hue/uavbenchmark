# UAV Benchmark A×L 责任定义字典

> **版本：** `axl-17A-ability-id-v2-2026-07-19`
> **范围：** 6 个 G 组、17 个 A 域、每个 A 的 L1–L4，共 68 个 A×L 责任单元。
> **用途：** 支持 Agent 在“同一业务案例”下切换目标能力等级，并据此生成不同的 Config、Task Template、实例、Trace / GT 和 Grader 预期。
> **转录方式：** 保留原 PNG 17 张能力卡片的责任语义，并应用团队于 2026-07-19 确认的 `ability-id-v2` 编号。

`LGA_含业务变量.png` 仍显示旧编号。当前机读编号以本文件为准：`A14→A5`、`A5→A6`、`A6a/A6b→A7a/A7b`、`A11→A8`、`A7→A9`、`A8→A10`、`A9a/A9b→A11a/A11b`、`A10→A12`、`A12→A14`；A1–A4、A13、A15 不变。

## 0. 核心使用规则

### 0.1 等级是累计式责任

| 等级 | 名称 | 通用责任边界 |
|---|---|---|
| `L1` | **域内潜在自主** | 人在任务回路中主导；系统提供局部理解、状态呈现、提示或单点建议。 |
| `L2` | **域内受令自主** | 人给出明确命令、结构或协议；系统负责一个执行片段或声明范围内的持续执行与维护。 |
| `L3` | **域内监督自主** | 系统负责名义任务过程的持续维护和异常识别；人在监督回路中承担确认、批准、纠偏或兜底。 |
| `L4` | **域内主导自主** | 系统在声明域内自主选择处置、完成恢复并验证结果；运行时兜底责任不再由人承担。 |

目标为 `A×L3` 时，Agent 不能只生成 L3 的新增行为；必须同时保留该 A 的 L1、L2 基础责任。目标为 `A×L4` 时同理，需保留 L1–L3。

### 0.2 同一案例切换等级时

- **保持不变：** 案例身份、核心业务目标、目标 A 域、主要业务对象和基本场景。
- **随等级改变：** 系统责任边界、人类角色、异常暴露方式、需引用的 JD、期望行为、Trace 和 GT。
- **不得自动发明：** threshold、业务规则、A×L 判定规则、simulator 接口或未声明的平台能力。

### 0.3 建议的 Agent 查询方式

1. 用 `a_id + level` 查询 `flat_a_by_l_index`。
2. 读取该项的 `inherits_levels`。
3. 将当前等级及所有继承等级的 `responsibility` 合并为完整责任集合。
4. 汇总其 `jd_refs`，再到 JD 业务变量字典中查询字段定义。
5. 针对同一案例只替换责任、输入暴露和证据要求，不随意改变案例业务身份。
