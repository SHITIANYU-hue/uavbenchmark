# Config Agent Pipeline 操作说明

本文是当前 `pipeline.html` 的正式操作文档。当前网页完成的是：

```text
选择并确认固定业务场景
  → 人工用一至三句话描述本次任务
  → Gemini 调用 1：扩充为完整任务文案
  → 人工修改并确认完整文案
  → Gemini 调用 2：提取变量并提出能力等级候选
  → 人工确认或修改能力等级
  → 复核 JD、TBD、外部依赖和自然语言 Task Template
```

当前网页还不运行真实 SUT、simulator、GT/Trace 或 Grader。

## 1. 启动页面

在项目根目录执行：

```bash
cd uavbenchmark
./scripts/start_agent_demo.sh
```

如果终端要求输入 `GEMINI_API_KEY`，粘贴后按回车。输入过程不可见，Key 只保存在本地服务进程中。

看到地址后，在浏览器打开：

```text
http://127.0.0.1:8765
```

不要直接双击 `pipeline.html`，否则页面无法调用本地 Agent API。

代码或场景配置更新后，应在旧终端按 `Control + C`，重新运行启动脚本，再刷新浏览器。

## 2. Stage 00：选择并确认业务场景

Stage 00 先固定业务场景，然后才允许输入任务。当前场景注册表包含：

| 场景 | 已登记的业务模块 |
|---|---|
| 高速巡检 | 任务规划、应急调度、检测识别、数据归档、飞行处置、事件联动、病害观测、进展预测 |
| 油气巡检 | 任务规划、应急调度、检测识别、数据归档、飞行处置、事件联动、进展预测 |
| 桥梁巡检 | 三维建模、分层航线设计、任务规划、检测识别、数据归档、数字孪生、事件联动、病害观测、进展预测 |
| 园区巡检 | 三维重建、航线设计、任务规划、检测识别、飞行处置、数据归档、事件联动、空地协同 |
| 其他巡检场景 | 河道、矿山、海事等；当前只固定规划、应急调度、检测和告警模块 |

操作方法：

1. 在“业务场景”下拉框中选择一个场景；
2. 阅读场景 Summary；
3. 检查“固定场景边界”和“可选任务模块”；
4. 检查业务变量表；
5. 点击“确认场景与变量”。

变量表中的绑定方式含义如下：

| 绑定方式 | 含义 |
|---|---|
| 固定值 | 该场景已经明确给出，Agent 只能引用，不能改写 |
| 固定取值域 | 场景只规定允许范围；本次究竟选择什么由任务描述或后续实例确定 |
| 由任务提取 | Stage 00 不填值，Agent 从人工任务描述中提取；没有证据时保持 TBD |

“可选任务模块”不等于本次正式评分能力。只有 Stage 02 人工确认稿中明确要求的责任，才允许进入 A×L 候选。

### “其他巡检场景”的额外要求

如果选择“其他巡检场景”，人工输入必须先说明具体子类型，例如河道、矿山或海事。没有子类型时，工作空间、对象、载荷、合规域和安全规则都应保持 TBD。

## 3. Stage 01：人工输入本次任务

这一阶段使用自然、简短的业务语言，不需要填写 A×L、GAL 或 JD 编号，也不需要人工编写完整 Task Template。通常 1–3 句话即可。

最少只需要说清楚“让无人机做什么”。如果业务人员知道，也可以补充：

1. 是否需要持续跟踪或持续更新；
2. 遇到异常时希望系统怎么做；
3. 最后希望得到什么结果。

可使用下面这种口语化骨架：

```text
让无人机【做什么】。
需要它【是否持续跟踪或更新】。
如果遇到【异常】，就【希望它怎么处理】。
最后给我【需要的结果】。
```

### 五个场景的任务输入示例

以下只是“人工输入示例”，不是固定答案，也不预先指定能力等级。

高速巡检：

> 让无人机巡查指定高速路段，持续识别和记录违法停车车辆。同一辆车短时遮挡后再次出现时不要重复上报；遇到定位异常先安全保持并通知调度人员，最后生成按路段整理的违停巡检报告。

油气巡检：

> 让无人机沿油气管道巡检并持续记录发现的异常。电量不足以安全完成任务时，由它自己决定缩短巡检、返航还是去安全点，并告诉我最后是否安全结束。

桥梁巡检：

> 让无人机检查桥底和指定结构件，把发现的病害标在桥梁三维模型上，并持续更新记录。不能确定的病害留给人工确认，不要直接给出维修结论。

园区巡检：

> 让无人机巡查园区道路和设施，发现异常后持续更新告警。需要近距离查看时通知机器狗去复核，最后把无人机和机器狗的结果汇总起来。

其他巡检场景：

> 这是河道巡检。让无人机沿指定河段持续查找异常漂浮物，发现后记录位置和图片并发出告警。遇到突发情况时给调度人员一个应急航线建议。

输入完成后：

1. 选择 Gemini 模型；
2. 点击“调用 1：生成任务文案”；
3. 等待页面进入 Stage 02。

本步骤是第一次 Gemini 调用，只扩充文案，不判断 A×L，也不提取 JD。刷新页面会按同一 Narrative Run ID 恢复，不会自动重新发起请求。

## 4. Stage 02：修改并确认完整任务文案

页面会展示第一次调用生成的七段式业务文案。此时应检查：

- 是否忠实表达了简短人工输入；
- 是否把仅属于场景的“可选模块”误写成本次任务责任；
- SUT、外部执行器、外部系统和人工责任是否分开；
- 是否编造了阈值、接口、业务规则或安全决定；
- 未确认的信息是否明确保留为 TBD 或待确认问题。

可直接在文本框中删除、补充或改写。点击“确认文案并调用 2：分类与提取”后，页面使用同一个已选模型再次调用 Gemini。这一次只做 A×L 分类、JD 提取和外部依赖识别，不允许改写已确认文案。

因此，一次完整演示会消耗两次模型请求：

1. 简短人话 → 完整任务文案；
2. 已确认任务文案 → A×L / JD 结构化候选。

## 5. Stage 03：人工复核能力与等级

这一阶段采用三层责任：

```text
Agent：理解自然语言并提出 A×L 候选
代码：检查候选是否属于受审目录、责任是否累计、必需 JD 是否齐全
人工：最终确认、取消或修改等级
```

页面中每项能力都有 L1–L4 下拉选项：

| 等级 | 复核时重点观察 |
|---|---|
| L1 | 当前结果、状态、风险或建议的单次输出 |
| L2 | 在 L1 基础上持续维护、持续更新或持续执行 |
| L3 | 在 L1、L2 基础上识别异常、执行允许的保护、上报并由人承担后续运行时决定 |
| L4 | 在 L1–L3 基础上自主选择处置策略、执行闭环并验证结果或安全终态 |

例如 Agent 判为 L2，但任务原文明示“异常后保护、上报并等待人决定”，应人工改为 L3。改级后，页面会按版本化目录补入该等级需要的 JD；没有值的字段仍为 TBD。

操作方法：

1. 勾选保留或取消不应进入正式评分的能力；
2. 用下拉菜单确认或修改等级；
3. 检查 Runtime Dependencies，确保外部执行器、外部系统和人工责任没有计入 SUT；
4. 必要时点击“人工补充外部依赖”；
5. 点击“确认能力等级”。

人工改级会保存为新的 Human Revision，不覆盖 Agent 原始候选，也不会再次调用 Gemini。

## 6. Stage 04：复核变量提取

JD 表展示：

- JD 编号与规范名称；
- 固定场景值或任务提取值；
- 证据/来源；
- `原文给定`、`候选` 或 `TBD` 状态。

检查原则：

- 固定场景值必须能追溯到当前场景注册表；
- 任务值必须能追溯到 Stage 02 人工确认后的完整文案；
- 人工填写的确定值必须同时填写确认依据；
- threshold、业务规则和接口没有来源时继续保持 TBD。

需要补充时点击“统一补录 TBD / 缺失 JD”。保存人工修订不会调用 Gemini。

## 7. Stage 05：确认自然语言 Task Template

最终正文应当是业务人员可直接阅读的任务叙述，不显示以下内部实现标签：

- A×L；
- GAL；
- `jd-x.x`；
- schema 字段名。

A×L、JD、来源和外部依赖保留在结构化结果中，正文只表达环境、任务责任、异常处理、外部边界、输出和完成条件。

如果正文出现内部编号、缺少必要段落或包含无来源规则，确定性校验会显示 `needs_review`。可进入人工审核工作台修订，然后保存新版本。

## 8. 运行记录

每次 Agent 运行和人工修订保存在：

```text
.agent_runs/
```

记录包含：

- 场景 ID 和版本；
- 简短人工任务原文；
- 第一次调用的扩充文案、Narrative Run ID 与人工确认稿；
- 第二次调用的模型名称、Classification Run ID 与结构化候选；
- Agent 原始候选；
- 人工修订版本；
- 校验问题和模型用量。

记录不包含 API Key，且 `.agent_runs/` 默认不提交 Git。

## 9. 常见问题

- 页面仍显示旧版流程：在终端按 `Control + C`，重新运行 `./scripts/start_agent_demo.sh`，然后刷新。
- 页面显示“等待本地服务”：确认打开的是 `http://127.0.0.1:8765`，并保持启动终端运行。
- 页面显示 Key 缺失：停止服务后重新启动并输入 `GEMINI_API_KEY`。
- Gemini 返回 503：当前模型高负载，稍后重试或手动改用备用模型；页面不会自动连续重试。
- 结果是 `needs_review`：这不是调用失败，表示候选需要人工处理。
- Stage 03 没有某个能力：先核对 Stage 02 的确认文案是否明确写出了该责任；不要仅凭场景的“可选模块”强行添加能力。
- “其他巡检场景”产生大量 TBD：先在任务开头明确河道、矿山、海事或其他子类型，并补充受审的对象、载荷和合规信息。
