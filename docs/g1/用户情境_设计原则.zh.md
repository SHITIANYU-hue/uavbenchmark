# 用户情境：模拟用户与可复现测试的设计原则

> 编号基线：本文已迁移到 `ability-id-v3-2026-07-20`，当前能力使用连续编号 A1–A17。
>
> 草稿 v0.3 · 定位「用户侧 config」这一目前缺失的一等抽象。
> 姊妹参照：Avant-Bench 流水线图（A 边坐标系）、内部前期资料《无人机案例到可执行 config 规范》（config 三分/承重不变量/AxL）、G1 人机交互组 L1–L6（难度轴）、E0–E5 评判阶梯（可复现强度轴）、pre/actions/post 三段踩点（真值骨架）。
> 术语约定（消歧）：**交互深度 = L0–L4**（难度，来自 G1；L0=该边不涉及）；**复现档 = R0–R4**（生成确定性，本文立，照 E0–E5）。二者正交。
> **术语（三分，勿混）**：
> - **用户模**（user model / `user_model`）：对用户建模 = `execution_anchors ⊕ persona`。（旧称"用户情境"，本文标题沿用；替换 config 三分文档字段 `user_membrane`——那是"模/膜"错配。）
> - **交互膜**（interaction membrane）：conversation 交互发生的面 = A1/A5 对话边。"膜"只指这个，不指 config。
> - **events**（event&trace）：交互 collapse 出的具体轮次。
> - 链条：**用户模** →(经 materializer 在)→ **交互膜**(A1/A5) →(collapse)→ **events** →(judge 投影 anchors)→ **GT**。
> 生成链（§7 起）：`template（只有 A×L）→ mock_user_constructor（+world+业务知识）→ config = execution_anchors ⊕ persona → materializer（R 档）collapse → event&trace → judge 投影出 GT`。
> 两相位真值（§5）：跑前 = `execution_anchors`（可声明规格，invariant）；judge 跑后 = 投影到 collapse 出的 trace → GT（判决）。整例见 [`G1示例_从AL到persona_config.zh.md`](G1示例_从AL到persona_config.zh.md)。
>
> **v0.3 增补**：§7 生成管线与边界、§8 逐维塑形规则表、§9 config 走查（L2/L3/L4/混档）、§10 materializer 运行时。原 §7–§9 顺延为 §11–§13。

## 0. 定位基准：A 边坐标系（先钉死坐标，后面全挂这上面）

一切从 harness 的**边**说起。每一个能力维 A 不是抽象标签，而是**标在 harness 的哪条边上**——即"这条信息/动作从哪进、往哪出"。这是整份设计的坐标系。

```
                 ┌─────────── 用户侧(G1 · 域不变)───────────┐
                 用户输入 / 模拟用户 agent            输出报告
                    │  ▲                                 ▲
                 A1 │  │ A5   ← 用户膜:我们的主战场       │
                    ▼  │                              评估流程
        ┌───────────────────────────────┐  埋点信息监测  (打分器/合约/门)
        │        Agent Harness          │─────────────────▶
        │   ┌─────── A3(agent loop)──┐  │
        │   │       被测对象 SUT      │  │
        │   │  A4 ◀──┐      ┌──▶ A2  │  │
        │   │        操作接口         │  │
        │   └───────────────────────┘  │
        └────────┬──────────────▲───────┘
             A10,A11│              │A6,A7,A8,A9   ← 世界侧(G2/G3 · 分域)
                  ▼              │
              数据集 / 仿真 / 真机测评场景
```

| A | 位置（哪条边） | 方向 | 流什么 | 归属 |
|---|---|---|---|---|
| **A1** | 用户 → harness | 入 | 用户下达 / 上下文 / 指代 | **G1 用户膜（域不变）** |
| **A5** | harness → 用户 | 出 | 请示 / 汇报 / 回显确认 / 交接 | **G1 用户膜（域不变）** ⭐ |
| A2 | 操作接口 → SUT | 入 | 任务规范 / 约束进 SUT | SUT 内（经端口） |
| A3 | harness ⇄ SUT 的 agent loop | 环 | 意图推理 / 任务解释 | SUT 内（loop） |
| A4 | SUT → 操作接口 | 出 | 交互到执行编排（下发动作） | SUT 内（经端口） |
| A6 / A7 / A8 / A9 | 世界 → 操作接口 | 入 | 感知 / 自定位 / 相对定位 / 健康 | 世界侧（G2，分域） |
| A10 / A11 | 操作接口 → 世界 | 出 | 导航航迹 / 飞行控制 | 世界侧（G2/G3，分域） |

> ⚠ **图上的一处标注纠正**：Avant-Bench 流水线图把右上（harness→用户）那条边印成 `A4`，按本文定义那条边是 **A5**（用户膜的返回边）；`A4` 是 SUT 内部 SUT→操作接口的编排下发。二者别混。

**三个关键判断（本文全部结论的地基）：**

1. **A1 + A5 = 唯一的用户膜，且域不变。** 这两条边就是「用户情境 / event&trace 用户」物化的**物理落点**：A1 上流"用户下达/应答"，A5 上流"agent 的请示/汇报/回显"。**D 域、C 域、meta 用的是同一对边、同一套 schema**——变的只是穿过它的值引用哪个世界。"人机交互跨域一样"在图上就是 **A1/A5 这对边跨域不变**。
2. **A2/A3/A4 是 SUT 内环，我们不碰。** 规范进→推理→编排出，对黑盒 SUT 不可见。做用户模拟**只在 A1/A5 上出招、在 A1/A5 上收 trace 判分**，绝不伸进内环。
3. **A5 是评测可判性的主来源。** 该请示时请示了没（A5 时序）、回显忠不忠实（A5 内容）、汇报对不对——全在 A5 采。A1 是"喂进去"，A5 是"判出来"。（这正是 config 三分文档案例二里 `echo_before_act`/`echo_faithful` 卡全挂 A5×L3 的原因。）

**出题流水线（从模糊到确定的最后一跳）：**

```
benchmark template  ──选 A×L 格──▶  变量实例化(场景/难度)  ──materialize──▶  event&trace 用户(落在 A1/A5)
   (抽象骨架)            (AxL 覆盖矩阵)                       (最后才具体 · R 档决定确定性)
```

A 的位置是最稳的骨架（harness 的边固定）；template 先挑"考哪几条边的哪个 L"；**最后才**把用户侧（A1/A5）物化成具体 event&trace——这是 §4「R0→R4 从模糊到确定」那把梯子的最后一跳：template/情境是模糊态，event&trace 用户是它 materialize 出来的确定态。

## 1. 问题：世界侧清晰，用户侧缺席

**世界侧 config 已是一等公民**：`scenario.py` 是 ground truth，`worlds.库` 是变体预设，`worlds.造(id)` 是构造入口，动词确定性地改变世界态。世界这一半可精确声明、可枚举、可复现。

**用户侧（A1/A5）没有对称物**。今天"用户"只是测试脚本里的字符串列表（`步骤: [("用户", 文)…]`）加硬编码字典（`上级脚本`），100% 静态、零反应性——不管 agent 在 A5 说什么，A1 上的下一句照发。后果：① **测不动最该测的交互路径**（求澄清、请示、变卦、L4 目标欠定→系统内化——都需要用户对 A5 输出**有反应**）；② **没有用户侧真值**（判不了"agent 有没有问出用户心里那口井号""该请示时请示没有""用户目标达成没有"）。

一次交互评测的 fixture 本该有**两半**：世界侧 + 用户侧。本文定义用户侧这一半 = **用户情境**。

| | 世界侧（已有） | 用户侧（本文提议） |
|---|---|---|
| 配置物 | `Scenario`（ground truth）+ `worlds.库`（变体） | `用户情境`（目标+私有知识+策略+预算）+ `用户库`（变体） |
| 提供什么 | 物理/工具环境 | 交互对手（A1/A5 上的人） |
| 驱动方式 | 动词改变世界态 | agent 的 A5 输出触发用户下一步 |
| 真值来源 | 世界终态 / 动作序列 | 目标达成 + 披露/澄清/请示/兜底序列 |
| 构造入口 | `worlds.造(id)` | `用户.造(id)`（待建） |
| 变体轴 | 世界变体 | 用户变体 |

**核心不变量（fixture 即 harness · 用户侧版本）**：用户情境**同时**是"用户行为生成器"和"用户侧真值来源"。私有知识账既驱动披露、又定义"该问出什么"的真值；目标既驱动 agenda、又定义"是否验收"的真值。二者是同一份 config 的两个投影，不许分叉。

## 2. 定位原则：测什么，由"用户注入什么/保留什么"钦定

**模拟用户建在某个真实世界 config 之上**。用户情境是世界情境的投影：给定世界（井、异常、禁飞区），

- **用户目标**引用世界对象（"让 W-05 巡检完并拿到照片证据"）；
- **私有知识账**把世界 ground truth 切成三份：`{agent 已知}`、`{用户已知·待披露}`、`{双方皆不知}`——这条切分线就是"信息不对称"的显式声明；
- **兜底/约束**引用世界规则（禁飞区、限高、低电返航阈值）。

**一条用户情境 = 一次能力探针的规格说明**。我们测的是 agent 的交互能力，**具体测哪条能力轴，完全由"用户把什么放进 A1、把什么留在自己心里、要求 agent 在 A5 上回应什么"决定**（这正是 G1 的判断公式，见 §3）。定位一个测试 = 填这张表：

```jsonc
用户情境 U = {
  基座世界:  <worlds.库 id>          // 建在哪个世界上
  目标:      <引用世界对象的验收谓词>   // 达成即通过；同时是 scorer 的"目标达成"卡
  私有知识:  {槽位 → 值}             // 只有被问到才披露；同时是"该问出什么"的真值
  显式约束:  [人此刻交代的约束/规则]    // L3 才大量出现；L4 退出
  显式兜底:  [人此刻交代的兜底规则]     // L3 显式给；L4 期望系统内化
  策略:      <遇 X 类 A5 输出如何反应>  // 遇请示/遇变卦/遇欠定…
  预算:      {最多轮数, 耐心阈值}      // 何时不耐烦、何时放弃
  终止:      目标达成 | 预算耗尽
}
```

**可控流程 = normcast 思路搬到用户侧**。normcast 的范式是"结构确定、表层由 LLM 自然化"。用户侧照搬：

```
用户情境 ──(确定性展开·可枚举)──▶ 交互流程骨架(agenda) ──(LLM 自然化·仅表层)──▶ A1 上的自然语言用户话
   │                                  │                                    │
 可声明·可复现·可对比的真值源        可控·可 diff·可当 GT              有波动·不进真值·只是膜
```

骨架是可控、可 diff、可当真值那层；自然语言是可波动、不进真值、只当膜那层。**真值绑在骨架上，不绑措辞**——与世界侧 scenario=ground truth、动词确定性、NL 只在表层，同一个模子。

## 3. 难度原则：交互深度 L0–L4 = 用户"注入 vs 保留上下文"的量

难度**不是**"用户话越简越难"，而是 G1 的核心判断：

> **交互输入形态 + 上下文是否由人显式提供 + 兜底规则是否由人显式提供 + 系统是否能内化 = 难度。**

反直觉但关键：**L3 的用户话可能比 L4 还长**——L3 人还得把任务上下文、约束、确认规则、兜底策略一条条显式交代（都压在 A1 上）；L4 人只给目标，其余全期望系统内化。难度在"上下文住在谁那里"，不在字数。

| 交互深度 | 形态 | 用户在 A1 注入的上下文 | 用户保留/期望系统内化 | A5 反应性要求 |
|---|---|---|---|---|
| **L0** 不涉及 | 该边无流量（不上场） | — | — | — |
| **L1** 动作级 | 单点飞控/单命令，全参数给全 | 无歧义、无上下文 | 无兜底闭环 | 基本不反应 |
| **L2** 受令 | 有边界命令/任务片段 | 指令进入，上下文很窄 | 兜底由人负责 | 少反应（顶多确认收到） |
| **L3** 监督 ⭐ | 目标 + **显式约束 + 显式确认规则 + 显式兜底** | 大量：任务上下文全交代 | 兜底策略需人说明/批准 | **必须反应**：回答澄清、批准/否决请示、异常确认 |
| **L4** 主导 ⭐ | 高层目标 / 审计请求 | 极少：只给目标 | 运行设计域/安全边界/兜底**全期望系统内化** | 反应稀疏但关键：可事后审计、可对越界喊停 |

**L0 = 该边不涉及**（没流量、没发生），是 per-A 向量的"零/关"；因不发生所以不测。注意它**不同于** `role=hidden`（发生了但不打分）：L0 ⇒ 不测，但不测 ⇏ 一定 L0。详见 §8。

**G1-ness = (A1 或 A5 非 L0)**：A1×L0 **单独不出 G1**——种子可从 **A2（底层操控激活/任务上传/调度触发）**、世界事件（A6 感知触发）或定时器进，人只在 A5 侧监督/接收，这正是高自主**"监督型 G1"**（L↑→A1↓、A5 仍在）。只有 **A1×L0 ∧ A5×L0**（两条人机边都空、无人在环）才真正出 G1，退成 G2/G3。种子的持有者随发起边而变：A1 发起→种子在 `user_model.events`；A2/世界事件发起→种子在 world config。

两条正交的难度旋钮：

- **反应性档位**（本文核心新增）：L1/L2 用户近乎无反应 → 静态脚本足够；L3/L4 用户**必须**根据 A5 输出决定 A1 下一句 → 需策略型/LLM 型用户（见 §4）。**信息不对称 + 反应性，是用户侧难度的主来源**。
- **动态触发深度**（借 config 三分文档 A4 的 `runtime_signals`）：用户/环境执行中**注入触发信号**（`电量=28%`、"临时加个 D 区"）是独立旋钮，对应用户情境里的 `预定扰动: [{触发, 内容}]`。

**轮数上限由难度决定，不由 R 档**：`轮数上限 = 1(种子) + 澄清上限(A1_L) + 确认上限(A5_L)`；A2 只加单轮载荷不加轮。峰在 L3（见 §10.3）。

**第三条正交轴：noise（realism/robustness）**——真实用户的脏（口语/错字/跑题/自我更正/掺无关背景/情绪）。与 A×L 难度、R 复现档**正交**：noise 扰的是 A1 表面话（主要压 A3 解释 + A1 消歧），**真值（私有知识/目标）不动 → GT 对 noise 不变**。同一 (A×L, GT) 可配不同 noise 档测鲁棒（见 §13.2）。

**难度可覆盖，不互斥**：达 L3 的用户应能被 L1/L2 的 agent 正确处理；L3 的新增门槛就是上下文、显式约束、显式兜底、及时的监督请求。

## 4. 可复现原则：从模糊到确定，复现档 R0–R4（照 E0–E5 立）

**先模糊探索、再固化复现**。E0–E5 是"评判"侧的消融阶梯；在"用户生成"侧立对称阶梯 **R0–R4**，从自然但不可复现，到确定可二次复现。R 档 = config 三分文档 `user_membrane.materializer`（本文更名 `user_model`）的取值（它只画了 `bundle`/`mock-agent` 两点，这里补全五点）。

| 复现档 | 名 | 机制 | materializer | 复现性 | 类比 |
|---|---|---|---|---|---|
| **R0** | 自由 | 一句人设，自由 LLM 扮用户 | mock-agent | 最低（高方差） | E0 naive |
| **R1** | 人设+目标 | 结构化人设+目标+隐藏信息的 LLM 用户（τ-bench 式） | mock-agent | 低（LLM 抖动） | E1/E2 |
| **R2** | 策略+LLM 膜 ⭐ | **确定性 agenda/策略核**决定说什么/披露什么/何时终止，LLM **只自然化措辞** | mock-agent | 语义确定、表层软 | 方案丙硬核+软锚定 |
| **R3** | 触发式规则 | 规则/判定门匹配 A5 输出 → A1 应答（关键词/`<是:…>`） | mock-agent | 给定 A5 即确定 | E3 strict |
| **R4** | 固定脚本/卡带 | 冻结转录 或 `(A5 消息 → A1 应答)` 卡带缓存 | bundle | 完全确定 | E3 字面 + 录制回放 |

**工作流方向 = 探索↓固化**：

```
   R0/R1 探索              R2/R3 固化               R4 冻结
  (LLM 用户放开跑，       (把跑出的确定性          (整段转录或卡带，
   发现有意思的交互流)     agenda/规则析出钉死)      当快照回归)
```

**固化 = 编诺棘轮**（`app/backend/normadapt.py`）：不自建固化机制，复用编诺的 `待确认 → 确认沉淀 → 守门子集回归 → 退化回滚`（详见 §6）。`加自会话`（`runner.py:424`）只是其一入口。等同 snapshot testing——快照因 agent 漂移失配时，回到 config 用 R1 重录。

**每条测试显式声明 R 档**（像 mode tag），于是"硬确定门（T2）还是含 LLM 膜的软门（T3）"一目了然——直接接现有 T1/T2/T3 分层。建议：**config 是活测试资产（R1/R2，语义活、守能力）+ 冻结转录是快照资产（R4，字面死、守回归），双件套**。

**推荐主力 = R2**：决定测试语义的部分（披露什么、何时不耐烦、何时终止、目标达成谓词）确定化，LLM 只把意图说成人话。复现性由确定性内核保证，自然度由 LLM 膜提供。

## 5. 真值原则：跑前声明 execution_anchors，judge 时投影成 GT（两相位）

真值分**两个相位**，别混：

```
① 跑前(可声明规格) : execution_anchors —— "一条正确运行必须办成/说清什么",invariant、persona 无关
② judge(跑后判决)  : 把 execution_anchors 逐条投影到 collapse 出的 trace → 命中/未命中 = GT
```

**跑前叫 `execution_anchors`（规格），跑后投影到塌下来那条 trace 才叫 GT（判决）**——同一份内容、两个相位（合仓里"先不翻，judge 时才对"）。§ 本节声明的"点"都是 `execution_anchors` 的字段；判分 = judge 时数命中（详见 [`G1示例_从AL到persona_config.zh.md`](G1示例_从AL到persona_config.zh.md)）。

复用 pre/actions/post 三段骨架（意图点/动作点/终态点），**新增一类挂在 A1/A5 上的"交互点"**。每类点 = 一个 `execution_anchors` 字段：

| 点类型 | 判什么（judge 时） | 采在哪条边 | execution_anchors 字段（跑前规格） | judged_by |
|---|---|---|---|---|
| 意图点（沿用） | agent 读对用户真实意图没（L4 欠定尤关键） | A1→A3 | `target` + `required_disclosures` | symbolic/lm_call |
| 动作点（沿用） | 工具调用对不对 | A4→世界 | `required_actions` | symbolic |
| 终态点（沿用） | 跑完世界/会话终态对不对 | 世界终态 | `goal_predicate` + 世界终态 | symbolic + 救援 |
| **澄清点** ⭐ | 规格含糊时 agent 该问是否问了（且没瞎补） | A5 | `required_disclosures` 的缺口（应问 X/不应瞎补） | symbolic |
| **披露/获取点** ⭐ | agent 是否问出了私有知识后再动作 | A5↔A1 | `required_disclosures` = 该获取清单 | quantitative(P/R) |
| **请示/确认点** ⭐ | 在正确边界请示/回显、按批复行事 | A5 | `required_confirmations` | symbolic |
| **兜底点** ⭐ | L3 遵显式兜底 / L4 未告知即内化 | A4+A5 | `bound_constraints`(L3) / `internalize_fallback`(L4) | symbolic(守卫) |
| **目标达成点** ⭐ | episode 结束用户目标达成没 | 世界+会话终态 | `goal_predicate` | symbolic/quant |

> 关键：`required_disclosures` 是"用户**必须传达**的内容"，但**用几轮、清楚还是含糊由 persona(P_π) 定**——W-05 可 turn0 直说、也可"上次那口"逼 agent 问。**anchors 定"要办成什么"，persona 定"怎么说出来"，GT 判"塌下来那条办成没"。**

### 5.x A → trace 踩点（anchor point）：每条边判分踩哪种步骤

把 §0 的 A 边接到 GT：**每个 A 在 trace 里踩的是不同种类的步骤**（踩点=定位+抽取，≠判断；相关方法来自内部前期资料《基于 Trace 的测评数据逻辑》）。这决定 execution_anchors 各字段去 trace 的哪里对：

| A | 边 | trace 踩点（踩哪种步骤） | 引擎对应（白盒 Mode A） | 该 A 判什么 | 域 |
|---|---|---|---|---|---|
| **A1** | 用户→harness | **意图理解的 `想`** | 读消息→`想`意图→`整合` summary | 读对用户真实意图没（对 `target`+`required_disclosures`） | 域不变（meta-V） |
| **A2** | 操作接口→SUT | **上下文参数的保存/处理步** | `记下`<键>(存约束/参数) + `查` | 约束/参数是否正确表示·存·取（`bound_constraints`） | 域不变（meta-V） |
| **A3** | agent loop | **后续持续的相关步** | `继续` + `整合` 产出的后续 simnorm | 推理/续做连贯、消解到位（意图链） | 域不变（meta-V） |
| **A5** | harness→用户 | **发消息 / 留存工作区的步** | `发`(用户/上级) + `记下`(留存/回传) | 请示/回显/汇报发对没、留存对没（`required_confirmations`） | 域不变（meta-V，留存目标分域） |
| **A4** | SUT→操作接口 | **对实体的具体编排步** | 域动词（航线飞/看/拍…=drone_command；C 域=改文件/跑命令） | 编排动作对不对、落对实体（`required_actions` on target） | **分域（domain-V）** |

两条推论：
1. **A1/A2/A3/A5 踩 meta-V 步（域不变），A4 踩 domain-V 步（分域）**——这正是用户侧域不变、只有 A4 触到域执行的原因（A4 = G1→G2/G3 的桥）。
2. **白盒 vs 黑盒**（U-Q7）：Mode A（simnorm 可见）→ A2/A3 的 `记下`/`继续` 直接踩；黑盒 SUT → A2/A3 内步不可见，只能踩**效果**（A2:动作没违约束；A3:序列合不合理）或 reasoning 字段。

承自 E0–E5 的纪律，用户侧照收：

- **分数由代码算**，不信 LLM 自报（`compute_score`）。目标达成、披露覆盖率、请示时机都是可代码判的谓词。
- **不可观测点不进分母**（`requires_semantic`/`cannot_verify`）。harness 建模不了的交互点显式排除、单独计"可见性缺口"，不污染分数。
- **单调救援**（E4/E5）：近义/方向判断只做 FAIL→MET。"披露内容语义等价"（"W-05" ≈ "5 号井"）走这条。
- **执行门（幻觉守卫）**：合约要求"先问出井号再飞"，trace 里 0 次澄清就直接飞 → 把"目标达成"从 MET 降 partial。用户侧的"没问就编"主防线。
- **目标达成对齐 jd-0.7**：逐 criterion 标 satisfied/pending/failed，overall 可 partial，**不谎报 success**。

**真值强度必须匹配 R 档**：R3/R4 给精确符号真值（E3 风格硬门）；R0/R1/R2 的交互点多需语义判（E0–E2 风格 LLM-as-judge，报告为主或软门）。即 **R 档（生成确定性）× E 档（判据强度）耦合**——一条测试的可复现强度取两者下确界。

## 6. 评测由运行沉淀而成（编诺即固化引擎 · 知行归一）

用户/评测侧**不自建固化机制**，复用编诺已跑通的人在回路棘轮。

**编诺 = 固化引擎**（`normadapt.py` Phase 1 + `lemma_induct.py` Phase 2）：

```
运行(行) → AI 评候选(有GT+结构过+裁判胜出+语义无不满足) → 待确认 → 人确认沉淀 → 守门子集回归 → 退化则回滚
                                                                     ↑ 正是 R0/R1→R2/R3/R4 的固化闸门
```

它已自带**单调守门**（新沉淀不许让存量退化 `跑守门子集`→退化 `corpus.删一条` 回滚），与 E4/E5 单调救援同精神。`提交候选`/`会话轮次` 已把口子开好：任何来源（`会话:<sid>`/手动）的好运行都能进同一 `compare_precipitate` 队列。用户侧只是再加一类源。

**知行归一 = 一次运行能同时沉淀成三样**。因为**一行简诺同时是计划（知）和 trace（行）**，`_canon()` 从 trace 抽的 canonical `::(…)` 串既是"发生了什么"又是"该怎么做"。于是一次被确认的运行一鱼三吃：

| 从同一条确认运行析出 | 沉淀成 | 已有机器 |
|---|---|---|
| agent 的 canonical simnorm | 语料（整合 few-shot） | `确认沉淀` ✅ |
| 高频动作子序列 | 派生引理（技能层，P3 卸载） | `引理归纳` ✅ |
| **用户 agenda + 交互真值** | **用户库变体 + 测试 GT** | **待建（本文）** |

**形成评测 = GT 自底向上沉淀，非自顶向下钦定（P2）**。传统 benchmark 先写 GT 再跑；编诺范式先跑、再把好运行确认成 GT。`kept` 语料就是被追认的金标；`lemma_induct._好simnorm()` 已把 `kept 沉淀 + 语料 + 通过 test_run` 一起当矿源，运行↔标准的边界本就在消融。

**缺的两个镜像件（编诺今天只给 agent 侧上棘轮）**：

1. **用户归一**（镜像 Mode B 的 `忠实归一`/`认知归一`）：把模拟用户的 A1 原始话**归一成"用户意图 simnorm"**（披露/追问/批准/变卦/催促/放弃… 作为用户侧动词），让用户行为进入同一套可评分、可沉淀的货币。没有这步，交互点真值只能靠字符串匹配。
2. **编用户 / 编评测**（镜像 `引理归纳`）：把**高频用户行为模式**抽象成**用户库变体**（§8 的 含糊型/变卦型/追问型不必手写，从确认的 episode 里挖），把**高频交互真值形状**抽象成**评测点模板**。

## 7. 生成管线：template → mock_user_constructor → config → materializer

### 7.1 四段管线与边界（各管各的，不越界）

```
template(需求方:只有 A×L)  ─┐
world config(世界真相)      ├─▶ mock_user_constructor ─▶ config(内核) ─▶ materializer(R 档) ─▶ event&trace
业务知识(SOP/常识)          ─┘        (三源 JOIN)          (声明式 policy)   (反应式跑)      (涌现的多轮)
```

| 层 | 只管 | 有没有 simnorm/交互脚本 |
|---|---|---|
| **template** | 考什么（A×L + 覆盖轴） | ❌ 一行都没有 |
| **mock_user_constructor** | 把 A×L 翻成用户内核 + 锚世界 + 灌常识 | ❌（声明式内核，非脚本） |
| **config** | 用户内核（目标/私有知识/策略/预算） | ❌（policy 是函数，非流程） |
| **materializer** | 反应式执行 policy | 只在**产出的 trace** 里出现 |

**科学性来自静态/运行时切分**，正好对称世界侧：

| | 静态（声明·可枚举） | 运行时（反应·跑起来） |
|---|---|---|
| 世界侧 | `Scenario` / `worlds.库` | `World.py`（动词反应式） |
| **用户侧** | **template → config** | **materializer（应答器·反应式）** |

由此三条**正交轴**（不缠在一起）：`template 参数化轴 (A×L, 场景, 用户变体) = 考什么` × `materializer R 档 (R0..R4) = 多确定` × `agent 行为 = 实际怎么走`。同一 template 可在任意 R 档跑。

### 7.2 template：只有 A×L（per-A 的 L 向量 L0–L4）

template 是**需求方**，只声明"考哪几条边的哪个 L + 在哪些变量上扫"，**不含内核、不含交互骨架**（交互形态是下游涌现的，写死反而扼杀考点）。它是一个 **per-A 的 L 向量**（`axl_cells`），每格各带自己的 L（L0=不涉及）：

```jsonc
template "D-G1-L2-受令闭环" {
  interaction_group: "G1",
  axl_cells: [                       // = per-A 的 L 向量
    { A:"A1",  L:"L2", role:"primary",    desc:"[用户→harness · 入 · 用户下达] L2:一条完整有边界命令" },
    { A:"A2",  L:"L2", role:"supporting", desc:"[操作接口→SUT · 入 · 约束进 SUT] L2:窄边界经端口进 SUT" },
    { A:"A3",  L:"L2", role:"supporting", desc:"[harness⇄SUT · agent loop · 意图推理] L2:显式命令浅解释" },
    { A:"A4",  L:"L2", role:"primary",    desc:"[SUT→操作接口 · 出 · 编排下发] L2:闭环片段 segment_closed_loop" },
    { A:"A5", L:"L2", role:"primary",    desc:"[harness→用户 · 出 · 请示/汇报/交接] L2:执行完汇报、不主动请示" },
  ],
  变量轴:  { 场景:"∈ worlds.库", 命令:"∈ {单点拍摄, 单点测温, 短航线}" },
  sampling:{ grid_over: ["场景","命令"] }   // = 覆盖账本
}
```

**desc 两段式约定**：`[边表示（§0，跨 L 恒定）] L?:<L 特化（随 L 变）>`。前半段是那条边永远的身份，后半段随 L 升降——升 L 只改后半段，不重写边。

**组合是常态**：真实 template 是**混档向量**（如案例二 `A1×L3 + A5×L3 + A9×L4 + A6×L2`）。统一 L2/L3/L4 只是向量的对角特例。

### 7.3 mock_user_constructor：三源 JOIN

```
mock_user_constructor(template, world_config, 业务知识) → user_model config
```

它是 config 三分文档 `config agent` 的**用户侧特化**，把抽象 A×L materialize 成具体内核。三源缺一不可：

- **template = 需求方**：只提 A×L 要求，不给内容；
- **world config = 现实**：提供可锚定的真对象（哪口井、几何），让目标/私有知识**指得到实物**；
- **业务知识 = 常识**：template 和 world 都没有的**合理值**（默认高度、什么算完成、单点任务几轮），让用户像个懂业务的人。

**结构域不变**（逐维塑形逻辑通用，见 §8），**填充域参数化**（world + 业务知识按域换）。核心逻辑 = §8 的逐维塑形规则表：拿混档向量逐格套 → 拼出 config。

### 7.4 materializer：控制循环 + 决策核（R 档）

materializer = **一条 R 不变的控制循环** + **一个按 R 档换的决策核**（唯一变的东西被隔离出来）：

```
控制循环(R 不变):  收 A5 输出 → 决策核.下一步(A5, 情境态) → 产一条 A1 → 更新态  until 目标达成|预算耗尽
决策核(按 R 档换):  R4 回放 events / R3 规则查表 / R2 策略确定内容·LLM 只措辞 / R1·R0 LLM 自由
```

> **落进 app = 引擎 self-host**（§12 point 2）：控制循环就是 `agent 等 → 开始用户回应 lemma → 发 A1 → agent 使用 → … → 等` 的交替；决策核（R1/R2）就是用户侧那次 `想`（persona 入框架、用户对话+anchors 入素材）。不需要新引擎——**共享的是引擎代码，不是上下文**。
>
> **★两进程原则（钦定）**：agent 与 user_sim 是**两个交替的进程，各有自己的上下文**，只在两点耦合——`发`（一方说给另一方的那句话过膜）+ `上下文 intake`（过膜的话进接收方上下文）。**不是所有上下文都过膜**：agent 内部（整合/继续/记下/ISO 态/trace）用户看不见；用户有**自己的上下文** = `用户对话`（只含 agent 发给用户的话 + 用户自己说的话）+ persona + 私有知识/已披露。用户侧的 `想` **只读用户对话，绝不查 agent 的统一 {上下文}**（否则用户"看穿" agent 内部 = 假交互，且上下文糊了会导致重复/漂移）。
>
> **★用户侧开发同样必须满足 P1–P3（钦定）**——模拟用户就是一个 agent，和正常 agent 一样需要自己的 查/收/发 动词层：
> - **P1 显式执行性**：intake/查自己的上下文/过膜投递 都是**显式动词步**（meta-002 原语 `用户收`/`用户查`/`用户说`，镜像 agent 的 使用备料/查/发），`开始用户回应` 展开成自足动词链 `用户收 @' 用户查×3 @' 想 @' 用户说`——整条用户回合进 trace、信息流经 `%{}` 显形，**编排不许偷塞 活指称/隐藏 Python 过程**。编排只留：预算/达成 gate + 对手进程（agent 轮）触发。
> - **P2 持续性路径**：用户侧资产（persona 库/anchor 库/meta-002 corpus/lemma）用中累积——U-Q5 用户归一动词、U-Q13 加 `整合` 都长在这个动词层上。
> - **P3 卸载**：用户 corpus 长大后，常见应答（确认/批准/简短披露）走直用模板、不烧 `想`。

关键澄清（避免"怎么只有一轮"的误解）：

- **`events` 只放无条件/定时投递**（种子下达 + `预定扰动`）；**反应性多轮不在 config，由 `策略` 运行时生成**，涌现进 trace。
- **"events 里有几轮" = R 档的函数**：R4 冻结整段脚本 / R3 种子+`on:` 规则 / **R2 只种子**（反应轮全由策略生成）/ R1·R0 只种子。低 R 把多轮固化进 events，高 R 留给策略。
- **轮数 ≠ R 档**：轮数由难度定（`= 1 + 澄清上限(A1_L) + 确认上限(A5_L)`，见 §10.3），R 档只管"这些轮冻结还是现生成"。同难度不同 R，期望轮数相同。

## 8. 逐维塑形规则表：(A, L) → config 字段（constructor 的核心）

constructor 拿到 per-A 向量后，**逐格套这张表**往 config 加/退字段。**L0 = 空（恒等元，不加任何字段）**，L1→L4 单调叠加。

| A（边） | L0 | L1 | L2 | L3 | L4 |
|---|---|---|---|---|---|
| **A1** 用户下达 | `user_model` 无种子；种子改从 **A2 激活/世界事件/定时** 进（仍是 G1 iff A5≥L1，见 §3） | 种子=单点命令；`私有知识={}` | 种子=完整有边界命令；`私有知识={}` | 种子=**欠定**；`私有知识≠∅`（藏槽）；+`历史` | 种子=**高层目标**；`私有知识`→少量高层偏好 |
| **A2** 约束进 SUT | `显式约束=[]` | 极简局部约束 | `显式约束=[物理边界]` | +`[任务上下文/确认规则]` | `显式约束=[]`（退出）→ 转 `应内化兜底` |
| **A3** 意图解释 | — | — | 浅解释 | 深解释（消解欠定） | 推断隐含目标 |
| **A4** 执行编排 | 目标无动作（纯理解） | 目标=单点下发 | 目标=单任务闭环+`完成判据` | 目标=多点/全 mission | +`预定扰动`（触发重排 runtime_signals） |
| **A5** 监管/交接/兜底 | 策略空（fire-and-forget，不报） | **单向汇报**：`策略.遇汇报→收到/记录` | 完成汇报：`策略.遇汇报→终止` | +**双向** `遇回显→确认`；兜底进 `显式兜底` | `策略.遇请示→推回`+审计；兜底进 `应内化兜底` |

> A3 主要在 SUT 侧、**耦合于 A1**（不独立塑 config），见下方耦合约束。

**兜底翻转（跨 L 的主线）**——兜底不随 L 消失，只**换持有者**；且**按维度分档**（混档下多档共存）：

```
某兜底维度在 L2 → 人实时管（config 里没有）
某兜底维度在 L3 → 进 显式兜底（人显式给 · 测 agent 遵守）
某兜底维度在 L4 → 进 应内化兜底（人退出 · 测 agent 无告知即内化）
```

混档 config 里 `显式兜底`（L3 维）和 `应内化兜底`（L4 维）**同时非空**，各装不同维度；`策略` 也随维度一半"披露/确认"、一半"推回"。**L4 命门：`应内化兜底` 各方面必须列全**（能源/空域/链路/感知/任务/安全），漏一维就测不出那维的内化。

**constructor 要守的耦合约束**（不是所有组合都自由）：

| 约束 | 说明 |
|---|---|
| **A3 ≥ A1** | 欠定深（A1×L3）→ 解释必须够深（A3≥L3）；不能 A1×L3 配 A3×L1 |
| **兜底分维度落档** | 每个兜底维度按自己的 L 进 人管/显式兜底/应内化兜底；L4 维必须列全 |
| **轮数上限 = f(A1_L, A5_L)** | 由交互两维算 `预算.最多轮`，不看世界侧维（如 A9） |
| **难度 overall ≈ max(L)** | 整题难度按最高格算；但 config **形状是 per-cell 拼的** |

## 9. config 走查：L2 / L3 / L4 / 混档（同一口井，逐档演）

### 9.1 L2（五格全 L2 · 一次交互多格覆盖）

一条**完整有边界**命令，顺五条边走一圈（A1 下达→A2 内化边界→A3 浅解释→A4 闭环执行→A5 汇报）。**用户只动 A1/A5**，A2/A3/A4 从 SUT 对这条命令的处理里判。

```jsonc
"user_model": {
  "based_on_world": "禁飞区演练", "interaction_group": "G1", "level": "L2",
  "materializer": "rule",                 // R3:L2 用户几乎不反应
  "覆盖": ["A1","A2","A3","A4","A5"],
  "目标": { "对象":"W-05", "动作":"拍摄", "完成判据":"拿到 W-05 证据帧" },
  "私有知识": {},                          // L2 命令自足（A3 浅解释前提）
  "显式约束": ["高度 60m","速度 5m/s","禁入 Z-2"],
  "显式兜底": [],                          // L2 兜底靠人
  "策略": { "遇请示":"简短放行『你决定』", "遇完成汇报":"收到即终止" },
  "预算": { "最多轮": 2 },                 // = 1 + 0(A1) + 1(A5)
  "events": [ { "t":0, "kind":"下达", "text":"飞到 5 号井,60 米高、5 米每秒,别进 Z-2,悬停拍一张发我。" } ]
}
```

每格点名 config 一个字段：A1→完整 events；A2→`显式约束`非空且写进下达；A3→`私有知识={}`；A4→`完成判据`明确；A5→`遇汇报→终止`+短预算。

### 9.2 L3（A1/A5 升 L3 · 欠定 + 回显确认）

下达从"飞 W-05 拍照"换成**欠定**"复查有问题那口井"→ 逼出非空私有知识 + 反应性策略，materializer 抬到 R2。

```jsonc
"user_model": {
  "based_on_world": "禁飞区演练", "level": "L3", "materializer": "policy+llm",  // ⬆ R2
  "历史": [ { "ref":"run#K", "摘要":"W-05 渗漏读数超标(疑似),W-03/W-07 正常" } ],  // ✚
  "目标": { "真实意图":"复查 W-05 的渗漏并回报", "完成判据":"结论+证据帧" },
  "私有知识": { "有问题那口井":"W-05", "依据":"run#K 渗漏超标", "重点":"渗漏" },   // ✚非空
  "显式约束": ["异常先提醒我确认"],          // ⬆ 物理边界→规则
  "显式兜底": ["低电按规则返航"],            // ✚非空
  "策略": {                                 // ⬆ 反应性 = 私有知识上的投影
    "遇澄清:哪口井": "披露 私有知识['有问题那口井']=W-05(+依据)",
    "遇回显:确认理解": "对则确认、偏则纠正到 W-05",
    "遇径直执行(未回显)": "不阻止(交 GT 扣 A5 回显点)"
  },
  "预算": { "最多轮": 6 },                   // = 1 + 2(A1) + 3(A5)
  "events": [ { "t":0, "kind":"下达", "text":"把上次那轮里有问题那口井再复查一遍,重点看渗漏。" } ]
}
```

一条种子 events → 跑出 ~6 轮（下达→agent 澄清→披露→agent 回显→确认→执行→汇报）；agent 若从 `历史` 自消解则更短——**不写死，才测得出 A1×L3 消解 / A5×L3 回显**。

### 9.3 L4（只给目标 · 兜底内化）

下达只给高层目标；`显式约束/显式兜底` 退出，兜底搬进 `应内化兜底`（各方面列全），策略从"喂"翻成"推回"。

```jsonc
"user_model": {
  "based_on_world": "综合园区", "level": "L4", "materializer": "policy+llm",
  "目标": { "真实意图":"A 区管廊隐患全面排查,产出结论", "完成判据":"覆盖 A 区+隐患清单+兜底记录" },
  "私有知识": { "偏好":"重点渗漏,但别漏结构裂缝" },   // ⬇缩水:高层偏好,非待披露清单
  "显式约束": [], "显式兜底": [],            // ✚退出
  "应内化兜底": [                            // ✚L4 命门:各方面兜底搬到期望侧(不交付 agent)
    "能源:低电返航 ≥30% 储备", "空域:禁飞区/限高", "链路:失联悬停并恢复",
    "感知:定位退化→保护+标不可信", "任务:覆盖率<95%→重拍", "安全:最小间隔/避障"
  ],
  "策略": {
    "遇澄清:问约束/兜底": "推回『你按规程判断』(L4 不喂→测内化)",
    "遇请示:重大决策": "才介入批复",
    "遇违背应内化兜底": "不实时拦,事后审计式记一笔"
  },
  "预算": { "最多轮": 4 },                   // < L3
  "events": [ { "t":0, "kind":"下达", "text":"今天把 A 区管廊隐患过一遍,下班前给我结论。" } ]
}
```

**config 交付面变薄、期望侧变厚**——正是 G1"L3 话比 L4 长"的 config 版：L3 厚在用户嘴上，L4 厚在 `应内化兜底` 里。

### 9.4 混档（案例二式：交互 L3、续航 L4）

`{A1:L3, A3:L3, A2:L2, A4:L2, A5:L3, A9:L4}`——**兜底按维度分档，两张兜底表共存**：

```jsonc
"cells": { "A1":"L3","A3":"L3","A2":"L2","A4":"L2","A5":"L3","A9":"L4" },
"私有知识": { "那口井":"W-05" },            // A1×L3
"显式约束": ["高度 60m","禁入 Z-2"],         // A2×L2
"显式兜底":   ["异常先提醒我确认"],           // A5×L3 维 → 显式给
"应内化兜底": ["能源:低电返航 ≥30%"],         // A9×L4 维 → 内化
"策略": {
  "遇回显:确认": "确认/纠正到 W-05",          // A5×L3:反应式
  "遇澄清:哪口井": "披露 W-05",               // A1×L3:披露
  "遇问续航/低电": "推回『你按规程判断』"        // A9×L4:不喂·测内化
}
```

同一用户，对"哪口井"L3 披露（喂）、对"低电怎么办"L4 推回（不喂）——**反应性本身也按维度分档**，单档 config 里看不到。

## 10. materializer 运行时补充

### 10.1 events = 种子 / 策略 = 反应 / 多轮在 trace

见 §7.4。config 静态只放无条件种子（+`预定扰动`）；反应轮由 `策略` 运行时生成，涌现进 trace。**config ≠ run**：一条种子 events 可跑出 N 轮（N 由难度定、由 agent 行为浮动）。

### 10.2 R 档决定 events 丰满度（不决定轮数）

| R 档 | events 装什么 | 多轮在哪 |
|---|---|---|
| R4 bundle | 整段冻结脚本（某次真跑录下来的） | 全在 events |
| R3 规则 | 种子 + `on:澄清→"W-05"` 罐头条目 | 部分在 events |
| R2 策略+LLM | 只种子 | 全由 `策略` 生成 |
| R1/R0 | 只种子 | 全由 LLM 生成 |

### 10.3 轮数上限 = f(A1_L, A5_L)

```
轮数上限 = 1(种子) + 澄清上限(A1_L) + 确认上限(A5_L)
```

| L | 澄清上限(A1) | 汇报/确认上限(A5) |
|---|---|---|
| L0 | 0 | 0（不报） |
| L1 | 0 | 1（单向汇报） |
| L2 | 0 | 1（完成汇报） |
| **L3** | **2** | **3**（汇报+确认往返，峰值） |
| L4 | 1 | 1~2（汇报+审计） |

**单向汇报从 L1 起**（不逼 A1 回应，只加 1 个 A5 事件）；**双向确认往返从 L3 起**（才加真·往返轮）。验算：L2/L2→2、L3/L3→6、L4/L4→4；**监督型 A1×L0∧A5×L1 → 仅 1 轮汇报**（种子在 world 侧）。**轮数峰在 L3**（欠定+确认双高），L2/L4 都低——config 侧的"L3 交互最密"。常数 TBD，用 pilot 校准。实际轮数 = 上限 × agent 行为（好 agent 短路澄清）。

## 11. 与 benchmark config 三分的接口（融合）

config 三分文档给的是**容器**，本文给的是 **A1/A5 那格的内核**。接缝很干净。

**① 用户模嵌进 config 的 user 格**（config 三分文档字段原名 `user_membrane`，本文更名 **`user_model`**）——它今天薄（`events[] + protocol + on:请示:* + 指代`，一张被动膜，无内核），融合后向下兼容它的 `events`/`bundle`：

```jsonc
"user_model": {
  "based_on_world": "<world config ref>",   // §2 定位：用户建在某个世界上
  "interaction_group": "G1",
  "level": "L3",                            // §3 交互深度(注入什么/保留什么)
  "materializer": "policy+llm",             // §4 复现档 R0 free-llm|R1 persona|R2 policy+llm|R3 rule|R4 bundle

  // ── 内核(它完全没有的部分)──
  "目标":   { "谓词": "W-05 巡检完且拿到照片证据" },
  "私有知识": { "井号":"W-05", "高度偏好":"80m", "预算":"不知道" },  // 披露驱动 + 披露 GT
  "显式约束": ["异常先提醒我确认"],           // L3 有；L4 退出
  "显式兜底": ["低电按规则返航"],             // L3 显式；L4 期望系统内化
  "策略":   { "遇请示":"查私有知识逐条披露", "遇变卦":"...", "耐心":"澄清>2次不耐烦" },
  "预算":   { "最多轮": 6 }, "终止": "目标达成 | 预算耗尽",
  "预定扰动": [ {"触发":"turn:3", "内容":"临时加个 D 区"} ],  // §3 触发深度

  // ── 兼容：R4 时退化成它的原样 ──
  "events": [ ... ], "protocol": "..."
}
```

它的"膜"是外壳，用户情境是装进 `materializer:mock-agent` 里的心——这也直接回答了它 §16 待确认 #4（缺有状态用户模拟器）、#8（`on:请示:*` 多次触发消歧）：mock-agent 有状态，是因为它有目标+私有知识+策略；消歧不靠通配符，靠**策略在私有知识账上的确定性投影**。

**② 交互点真值 = 它的 `invariants[]` 卡（最干净的融合，不需新机器）**。§5 五类交互点逐一就是一张承重不变量卡，而它的**案例二（G1 主考）已原始地在做**：

| 本文交互点（采于） | 它的卡（案例二已有 or 新增） | judged_by |
|---|---|---|
| 请示时序点（A5） | `echo_before_act`(回显@'等确认@'执行) ✅ | symbolic |
| 回显忠实点（A5） | `echo_faithful`(回显与真消解一致) ✅ | lm_call |
| **披露/获取点**（A5↔A1） | `problem_subset`(筛问题子集 P/R) ✅ → 推广成"私有知识=该问出清单" | quantitative |
| **澄清点**（A5） | 新增 `ask_when_ambiguous`(该问问了/没瞎补) | symbolic |
| **兜底点**（A4+A5） | 新增守卫卡 `follow_fallback`(L3遵显式/L4内化) | symbolic |
| **目标达成点** | 新增 `goal_reached`(目标谓词 over 终态) | symbolic/quant |

融合动作：**把用户情境内核的目标/私有知识/兜底，自动物化成几张 `invariants[]` 卡**，喂进它跑通的三类 grader、AxL 聚合、防重复路由。E 档 = 卡的 `judged_by`，R 档 = `materializer`，取下确界。

**③ 用户持有的 hidden GT（它的模型缺这个）**。它的 `hidden_gt` 只有**世界持有**的真值。但**私有知识账是第三种 hidden GT——用户持有**：既驱动披露、又定义"该问出什么"的真值（fixture 即 harness 落到用户侧）。`指代表` 其实是用户私有知识泄进世界侧的产物，应归拢进 `private_knowledge`。

> hidden GT 有两个持有者：**world 持有**物理真值（评分时取）、**user 持有**私有知识（被问才披露，未披露部分即披露 GT）。

**④ 域不变性主张**：`user_model`（用户模）schema **域不变**（= Meta-V / G1 层），定义一次，插进 D/C/meta 任一 world config。用户目标/私有知识**引用**世界对象（drone 里是井号、coding 里是文件路径），所以**域参数化、结构域不变**——正如 Meta-V 动词域不变、只作用于不同 world。这推翻"每个域重写用户模型"：这份 D 域文档的用户模和 C-001 该用的**是同一 schema**，变的只是引用的 world 和填进私有知识的值。

**⑤ 出题流水线接上**（§0 的落地）：`user side template`（@Agent 团队）→ config agent 标准化 → `user_model` → **materializer 按 R 档物化**成 A1/A5 上的 event&trace 喂 SUT。这就是 §0「template → A×L → event&trace 用户」的完整链路。

## 12. 落到现有架构（不动手，先记边界）

对称性看清后，改动面很小：

1. **用户情境 fixture** = `Scenario` 的镜像；**用户库** = `worlds.库` 的镜像（`配合型 / 含糊型(逼澄清) / 变卦型(逼改+重整合) / 追问型(逼机制一多转) / 沉默上级(逼等) / 越权诱导(逼拒绝)…`）。**用户变体 × 世界变体 = 新的 benchmark 二维轴**，现有"变体组"概念直接吃下。
2. **用户方 = 引擎 self-host `上下文框架下想`（不是外挂 responder）** ⭐。加一个 `用户方 = 人 / 模拟用户` 开关（镜像 `应答方`）。**模拟用户模式**下，agent 落 `等` 不再挂起等人输入，而是自动触发 **`开始用户回应` lemma**（= agent 侧 `开始回应` 的镜像）：

   ```
   开始用户回应(对话, persona, anchors):
     ::(上下文框架下想: &[回应agent,对话], persona框架, anchors素材 | 下一句)   // persona 入框架, anchors 入素材
     @' <进度分流: 下一句 | 继续, 达成, 放弃>            // goal_predicate 达成? 预算(§10.3)尽?
     @' [::(发: 下一句, →agent)] @<继续>               // 发即 A1 incoming → agent 使用 → …→ 等 → 循环
     @' [::(停)] @<达成|放弃>
   ```

   于是**编排(materializer) = 两个 self-host lemma 经 `等`/`发` 交替**：`agent …→等 → 开始用户回应→发 A1 → agent 使用→…→等 → …` 直到 `goal_predicate` 达成或预算耗尽。**大半是现成件**：`想`/`框架下想`/`上下文框架下想`（生成，纯）、`等.py`+`stream_resume`（挂起/唤醒）、框架基础设施（persona 落 per-session 框架）、`scenario.py:24 触发oracle`（"agent 发到某渠道→触发对侧回应"的挂点雏形，把"查脚本 dict"换成"跑 上下文框架下想"）。`上级 = 策略更简单的 persona`，`上级脚本` 退役。
   **坑**：用户侧 `想` 与 agent 侧 `想` 须用不同模型/key（U-Q3 隔离，防同模型作弊）；第一版用户侧只跑 `想`（生成一句）不执行动词，以后再升成完整"用户简诺"（U-Q5 用户归一动词表）。
3. **test schema 加一档**：battery case 里 `步骤`（R4 脚本）与 `用户`（R1–R3 config）二选一；`跑一条` 分支——脚本模式照旧，用户模式走循环驱动。**存量用例零改动**。
4. **断言侧顺延**：期望/禁出/sim检查/态检查 全照用，新增 `用户侧检查`（目标达成、澄清次数、披露覆盖、请示时机）→ 物化成 `invariants[]` 卡。
5. **固化管线已在**：编诺 `compare_precipitate` 队列 + `确认沉淀`/守门/回滚 + `加自会话` + `反馈`同意轮提 GT，就是 §4/§6 固化的落地通道；只需加 `用户归一`。

6. **Meta-002 = 用户模拟域（钦定）** ⭐：`开始用户回应` + 用户归一动词不进 base meta，而是作**一个 ISO 域** `meta-002`（Meta-001 agent 膜的镜像），走现成域插件机制（`{name}_fixtures/ + bridge.接入 + ISO域注册`）。
   - 包 = `as_harness/meta_002_fixtures/`：`functions/_展开式lemma.py`（`开始用户回应` lemma）+ `externals/{world 用户世界, bridge 接入, scenario 用户情境}`（用户世界=有状态:persona/anchors/预算/已披露/达成）。**复用 base meta 的 `上下文框架下想/想/发/等`，不重造。**
   - **域不变、可叠**：与 D-001/C-001 共存（`世界s` 里 用户世界 与 agent 世界并列），anchors 引用 agent 域对象。挂 meta-002 = 开模拟用户模式。
   - 纯 additive：`_挂用户模拟` + `ISO域注册` 加一项，Mode A/D-001 一字未动。

7. **`等` 的 `来源` 三分（区分"两种等"）** ⭐：`等` 只有一个动词，但 `来源` 分了唤醒责任方——**meta-002 只收 `来源==用户`（+`上级`），系统/世界的 `等` 一概不碰**：

   | `等` 来源 | 语义 | 谁唤醒 |
   |---|---|---|
   | **用户**（缺省） | 等 A1 用户输入 | 人 / meta-002 `开始用户回应` |
   | **上级** | 等 escalation | 上级 persona（简化 meta-002）/ 触发oracle |
   | **系统/世界** | 等 world 事件 / async / 协程 park | 引擎/世界/协程（`边飞边看` 那类），**meta-002 skip** |

   最小改：`等.py` 加 `会话态._等来源 = 来源 or "用户"`（缺省=用户，保持现状）；meta-002 循环 gate `and 会话态._等来源 == "用户"`。记号：`::(等)`=等用户 · `::(等:{条件},{系统})`=等系统。

## 13. G1 统一 benchmarking pipeline（bootstrap：mock → 交互 → 筛选 → 固化）

前面各节（template/constructor/config/materializer/真值/编诺）串起来，就是一条**域不变的 G1 出题流水线**。它是 §4「探索↓固化」+ §6「编诺」的具体化：先用宽松 mock 跑出交互，再按效果和 trace 筛出 test 资产。

### 13.1 六步

```
① template(A×L 向量 + 覆盖轴)                                     ← 需求方,定考什么
② mock_user_constructor: 逐维塑形(§8) → mock user =
     { prompt(人设/目标) + 上下文(私有知识+world+业务知识) + noise }   ← R1 宽松 mock
③ run: mock user ⇄ simnorm/SUT → event&trace                      ← "和 simnorm 交互"
④ filter(编诺机制): 效果门(目标达成+目标格行为触发) + trace门(结构健康) + 人确认
⑤ precipitate:  GT(锚于②的私有知识/目标,经④验证/enrich)
                + 事实用户对话轮(R4 冻结 bundle / 或抽成 R2 策略)
⑥ 归档 test 资产 → benchmark 套件(hidden,judge 用)               ← 最终目标 = 形成 test
⑥'(可选副产品) 好 agent simnorm → 编诺 corpus(agent 可见)         ← 与 test 池防火墙隔离
```

跨域同构（G1 层），drone/coding/meta 同一条，只换 world + 业务知识。②③④⑤ = 上面"先 mock 一个 user agent（prompt+上下文+noise）→ 和 simnorm 交互 → 按效果和 trace 筛"的落法。

### 13.2 四条护栏（"benchmark" vs "跑着看"的分水岭）

1. **mock 必须被 template A×L 塑形**（定向，非 free-floating）：prompt 由 §8 逐维塑形出，瞄准具体 (A,L) 格；否则 mock+filter 一堆交互却不知覆盖了哪格，退化成"跑着看"。
2. **noise 只在表层，GT 对 noise 不变**：noise = 真实用户的脏（口语/错字/跑题/自我更正/掺无关背景/情绪），是**独立的 realism/robustness 轴**（与 A×L、R 档正交，见 §3），主要压 A3（解释）+ A1（消歧）；扰的是 A1 表面话，`私有知识/目标`（真值）不动 → 同一 GT 可配不同 noise 档。
3. **GT 锚在 mock 自己的 config，不靠 trace 无中生有**：核心 GT 本就在 mock 的 `私有知识`(披露 GT)/`目标`(达成 GT)/`应内化兜底`(内化 GT) 里（fixture 即 harness）；④是**验证 + enrich**，不是让 agent/trace 定义 GT（否则 GT 循环）。
4. **隔离**：mock user 的 LLM ≠ SUT 的 LLM（U-Q3），防同模型"心有灵犀"作弊。

### 13.3 test 形成 ≠ 编诺（机制同源、产品对立、可见性相反）

⑥ 的最终目标是**形成 test（度量资产）**，不是 corpus。它与编诺**共用固化机制，但产品/目的/可见性都不同**：

| | 编诺（corpus） | **test 形成（本 pipeline）** |
|---|---|---|
| 产品 | 语料（整合 few-shot）+ 引理 | GT + 冻结对话轮 + grader 绑定 |
| 目的 | 让 agent 变强（**aid**） | 度量任意 SUT（**exam**） |
| 消费者 | 整合，runtime | evaluator，judge time |
| 可见性 | agent **可见**（given） | agent **不可见**（hidden GT） |
| "守门"含义 | 新 corpus 不退化存量（护 corpus 质量） | test 还能否有效区分好坏 SUT（validity） |

**防火墙（实操命门）**：**test 的 GT/冻结轮绝不能回流进 agent 的 corpus**，否则"教到考题"（contamination），度量作废。同一次确认运行可喂两边（知行归一，§6 三投影里的两项），但产物落进**两个可见性隔离的池**：

```
好运行 ──┬──▶ 编诺 corpus(agent 可见,帮它)   ┐
         └──▶ test 资产(GT hidden,考它)      ├─ 两池,防火墙隔离(接 config 三分 hidden GT 线)
                                              ┘
```

**两个独立飞轮**：corpus 飞轮（§6 编诺，让 agent 越跑越强）+ **test 飞轮**（让 test 越跑越准）。test 的质量闭环 = **校准（Kappa）/ 红队 / 验诺**（config 三分文档 §14），不是编诺守门——"守门"这个形状转移了，但对象从"护 corpus"换成"护度量有效性"，正确的家在评测器侧。

### 13.4 双件套产出（接 §4）

同一条 template 产出两种资产：**R1 live**（②③，LLM mock+noise，探索态，方差大/自然/测鲁棒，T3 软门）→ 经④编诺机制 → **R4 frozen**（冻结对话轮，回归，T2 硬门）+ 可选 **R2 策略**（抽出确定性反应核）。
