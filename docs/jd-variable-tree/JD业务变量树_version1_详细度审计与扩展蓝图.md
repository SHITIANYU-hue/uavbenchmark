# JD 业务变量树 version1：详细度审计与扩展蓝图

> 日期：2026-07-22
> 状态：研究与实施蓝图，不是正式 JD 字典  
> 适用范围：当前 version1 中的 A1–A11  
> 目标：判断其他能力是否需要达到 A11 的细分程度，并给出可审计的扩展顺序、证据要求和边界。

## 1. 结论

可以继续把 A1–A10 展开得更细，但不应把“每棵树达到 75 个变量叶子”设为验收目标。

A11 当前叶子较多，主要有三个原因：

1. 除三个 canonical JD 外，还增加了 `PROPOSED-jd-11.4 控制执行条件与动态包络`。
2. `PROPOSED-jd-11.5` 同时承载近障 / 受限空间的世界几何、动态关系、派生控制压力和 A11 要求绑定。
3. 控制接口、运动包络、执行机构约束和近障场景本身都可以继续分成独立变量。

不同能力的合理叶子数量不会相同。判断是否应该继续拆分，应看一个候选项是否同时满足：

- 有独立定义，不是另一个变量的取值档位；
- 有独立的值类型、候选值域或引用对象；
- 有独立的启用条件、配置归属或可见性；
- 对 Seed 实例化、Config 投影、覆盖分析或审计有实际作用；
- 能找到本地资料、标准、论文、数据集或官方文档支撑；
- 不重复 global JD，也不把 A×L 等级当成 JD 取值。

因此建议采用“证据驱动的完整度”而不是“叶子数对齐”。A6–A10 与巡检业务、定位、导航和控制直接相关，适合优先补齐；A1–A5 可以继续细化，但应避免把 SUT 内部思维过程、persona 或私有推理链误做成业务变量。

本轮已先完成三个结构性更正：A6 补纹理与局部对比条件、A11 补一障碍物多通行空间模型与外观条件引用、页面补全 `jd-0.1`–`jd-0.10` 共享业务变量树。其余 A1–A10 内容只保留为逐能力审阅候选，不再批量写入变量树；只有能够对应现有上位节点并获得直接来源支持的独立变量，才会在后续审阅中落地。

## 3. `PROPOSED-jd-11.4` 的依据与确定性

### 3.1 它不是权威字典原有槽位

当前 66 槽位字典只定义：

- `jd-11.1 控制质量先验`
- `jd-11.2 控制工作机制`
- `jd-11.3 控制处置策略适用集`

因此 `PROPOSED-jd-11.4` 必须保持：

- `canonical_slot: null`
- `review_status: proposed`
- `derivation_status: proposed`
- 不计入当前 66 个 canonical JD

“控制执行条件与动态包络”这一名称是对多份资料的保守归纳，不是某一标准中的原文术语。

### 3.3 从资料到变量树的归纳过程

资料原文能够支持“这些维度客观存在”，但不能直接证明本项目应采用某个固定编号或阈值。因此当前拆分为：

```text
PROPOSED-jd-11.4  控制执行条件与动态包络
├─ 11.4.1  平台与载荷执行基准
├─ 11.4.2  指令接口执行约束
├─ 11.4.3  运动与姿态动态包络
├─ 11.4.4  执行机构约束
└─ 11.4.5  包络修正与适用条件
```

其中：

- “接口、更新、超时、速度、姿态、执行机构限制”属于资料支持的变量维度；
- “11.4”编号和“控制执行条件与动态包络”名称属于待确认建议；
- 所有具体 Hz、ms、m/s、m/s²、角度、角速度和余量阈值保持 TBD；
- 平台静态尺寸与性能仍归 `jd-2.2`，11.4 只引用；
- 健康、电量和实时资源状态归 A9；
- 航路与几何轨迹选择归 A10；
- 安全阈值、返航、降落和强制安全态归 A16；
- 11.5 描述世界侧近障场景，11.4 描述该平台在场景下可执行的控制包络。

## 4. 统一扩展方法

每棵树后续扩展时采用同一流程：

1. 从 canonical JD 出发，先拆“变量维度”，不先填值。
2. 对照 A×L，只提取责任所需的场景变量，不复制能力等级。
3. 检查是否已经由 `jd-0.x`、其他 A 或平台档案定义。
4. 为每个候选叶子补齐值类型、单位、激活条件、难度方向、配置归属和可见性。
5. 公开资料只支持变量存在、不支持值域时，值域写 TBD。
6. 只有当两个叶子在 Seed 采样、配置投影或审计上需要独立处理时才分开。
7. 最后再分高速巡检、园区巡检和跨场景候选值，不复制整棵树。

## 16. 验收建议

不要求所有能力与 A11 具有相同叶子数。建议改用以下验收问题：

1. 每个 canonical JD 是否已经覆盖其定义中的所有独立变量维度？
2. 每个叶子是否能被 Config Agent 检索、被 Seed 启用或被审计使用？
3. 每个值域是否有来源，或者明确标为 TBD？
4. 是否混入其他能力、global JD、A×L、自主等级或 simulator 接口？
5. 是否存在只为了增加树深或叶子数而拆出的无意义节点？
6. 高速巡检和园区巡检是否可以复用结构，仅通过候选值和启用条件区分？
7. 世界侧、用户侧、SUT 输入、Harness 和 Hidden GT 是否保持分列？

只有在这些问题均得到回答后，叶子数量才有比较意义。
