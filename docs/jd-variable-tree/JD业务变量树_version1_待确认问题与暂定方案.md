## 2. L0 的暂定数据合同

已确认语义：某能力在任务中出现，但该能力不采集、不评分且不计入覆盖。

本版暂定：

- L0 不属于当前 AxL 字典中的正式责任等级。
- L0 不写入 JD 叶子的 `value_domain`。
- L0 不写入 `used_by_axl`；该字段仍只允许引用正式的 L1–L4 单元。
- 在 episode、template 或 coverage 记录中另存 `coverage_level: L0`。
- Config Agent 检索到 L0 时可以保留能力上下文，但不得因此启用采集项、评分项或覆盖计数。

### 待确认

- L0 的正式字段名用 `coverage_level`、`collection_level` 还是其他名称。
- L0 是否需要保留最小运行日志，以便解释“为何不评分”。
- 统计报表中 L0 是否显示为“出现但未纳入评测”，还是完全从覆盖矩阵隐藏。

## 3. 决策 5：配置归属与数据投影分列

已接受。本版不再用一个字段同时表达“谁拥有变量”和“变量送到哪里”。

| 字段 | 暂定含义 | 当前候选 |
|---|---|---|
| `configuration_side` | 业务变量的配置责任 / 归属 | `world`、`user`、`shared`、`TBD` |
| `projection_targets` | Seed 实例化后投影到的配置或输入面 | `world_config`、`user_config`、`sut_input`、`harness`、`TBD` |
| `visibility` | 信息对 SUT、Fixture、Grader、Hidden GT 的可见性 | 沿用现有数组 |
| `observation_channel` | 评测时从哪里观察该变量或其效果 | `sut_input`、`sut_trace`、`fixture`、`grader`、`hidden_gt`、`TBD` |

暂定原则：

- A1 和新 A5 通常会投影到 `user_config`、`sut_input` 或 `harness`，但不能据此断言所有细节都由用户侧拥有。
- A2、A3、A4 多数属于 SUT 内环或任务合同，重点通过 `sut_input` 与 `sut_trace` 观察；不能因为它们位于交互能力组就全部塞入 `user_config`。
- `shared` 只表示配置归属可能由双侧共同确定，不表示双方看到完全相同的信息。

### 待确认

- `harness` 是否还要拆为 `fixture_config` 与 `grader_config`。
- 一项变量允许同时投影到世界侧和用户侧时，两个投影是否必须保持同值，还是允许各自保存不同视图。
- `sut_input` 是配置投影还是运行时消息通道；若两者都需要，应进一步拆字段。

## 4. 决策 6：推导状态与证据状态分列

已接受。本版增加 `derivation_status`：

- `given`：资料明确给出。
- `derived`：由一份或多份资料保守归纳。
- `proposed`：待确认建议。
- `TBD`：资料不足，暂不确定。

`evidence_status` 继续表达证据强度：

- `authoritative_existing`
- `source_supported`
- `inferred_candidate`
- `team_material_only`
- `TBD`

暂定原则：`given` 不自动等于“正式权威”。例如 A11 下的白墙窄缝示例赋值是 v0.6 团队草案原文，因此可以是 `given + team_material_only`；从该示例抽象出的跨巡检 Profile 则标为 `proposed + inferred_candidate`。

### 待确认

- 是否需要增加 `conflicting_evidence`，专门表示多份来源互相冲突。
- 一个节点同时含 given 和 proposed 候选值时，节点级状态取最高证据、最低证据还是 `mixed`。本版保留值级状态，节点级采用保守汇总。

## 5. 决策 7：persona 不作为 local JD 变量树的同层分支

暂定方案：

- `persona` 是用户模拟器或交互膜的 setup 资产，主要描述“怎么说、怎么回应”。
- persona 可以引用 `jd-0.4 在场人角色`，但不复制为 A1、新 A5 的 local JD taxonomy。
- `execution_anchors` 负责不随 persona 改变的任务事实和交互要求；persona 只改变表达路径和交互纹理。
- 如果 persona 的某个属性改变了任务目标、约束或判定，它已不再只是 persona，应回投到相应 JD 或 setup 合同。

### 待确认

- persona 库由 JD Catalog、Config Agent 还是独立用户模拟器模块管理。
- persona 轴中哪些属于纯表达纹理，哪些会改变信息披露策略，是否需要单独的可审计字段。
- persona 与 `jd-0.4` 的正式引用方式。

## 6. 决策 8：指称对象目录引用业务对象树，不复制 taxonomy

暂定方案：

- `jd-1.1 指称对象目录` 描述“哪些对象可以被指称、ID 如何组织、如何关联历史”。
- 具体巡检对象类别、病害类别、车辆类别仍由新 A6（旧 A5）业务对象感知树提供。
- A1 通过引用对象目录或 registry 取得对象类型，不在 A1 下复制道路病害、车辆、建筑等完整 taxonomy。
- 场景对象目录可由 `business_scenario_registry` 或后续专门 registry 提供。

### 待确认

- 对象目录的 canonical registry 是哪个文件或服务。
- 一个对象 ID 的唯一性范围是单任务、单场景、单次运行还是跨运行。
- 历史对象合并、别名和对象消歧由谁维护。

## 7. 决策 9：`jd-2.2 平台参数表` 只保留静态平台档案

暂定方案：

- `jd-2.2` 保存任务规范需要引用的静态平台类型、物理 / 运动学参数、载荷能力和设计运行域。
- 电量、健康、实时资源余量和运行中退化属于 A9（v1 旧 A11），不复制进 `jd-2.2`。
- 具体平台数值必须来自平台说明书、团队受审平台表或 simulator 正式文档；本版不补数值。

### 待确认

- “静态能力上限”和“当前允许包络”的边界。
- 平台参数表与 A11 控制执行包络候选字段的去重规则。
- 目前是否已有正式平台 profile schema。

## 8. 决策 10：A4 人机交接点与新 A5 交接协议分界

暂定方案：

- `jd-4.2 人机交接点`：描述任务流程中的“在哪里交接”，包括流程位置、有效窗口、暂停状态、状态快照和恢复位置。
- `jd-5.1 交接协议表`：描述“如何交接”，包括交接类型、角色权限、优先级、载荷、等待、确认、回读和恢复协议。
- 两者通过引用关联，不重复保存同一变量。

### 待确认

- 交接点与交接协议是一对一、一对多还是按场景动态绑定。
- “有效窗口”归 A4，“延迟预算”归 A5 的划分是否接受。
- 安全暂停状态是否应引用 A16，而不是由 A4 自己定义。

## 9. 决策 11：三类重规划 / 处置边界

暂定方案：

- A4：任务级重编排，改变阶段、步骤、技能组合或任务目标实现路径。
- A10（v1 旧 A7）：航路 / 轨迹级重规划，改变航路、航段或几何轨迹。
- A16（v1 旧 A12）：安全包络和反射级兜底，处理即时安全保护与处置。

判断顺序暂定为：先看“变更对象是什么”，再看“决策时间尺度”，最后看“是否属于安全反射”。一个事件可以同时触发多个能力，但每个变量只在其主责任树定义，其他树用引用表达。

### 待确认

- “返航”“等待”“改高度”在不同触发条件下分别归 A4、A10、A11 或 A16 的判定表。
- 多层策略同时触发时的优先级和审计记录。
- 哪些处置是任务策略，哪些是平台固有保护。

## 10. 决策 12：G1 树跨场景复用

暂定方案：

- A1–A5 采用跨场景结构，`applicable_scenarios: [cross_scenario]`。
- 高速巡检、园区巡检、油田巡检等差异尽量通过对象 registry、候选值、启用条件和配置投影表达。
- 不为每个场景复制整棵 G1 树。
- 对确实只在某场景成立的候选值，在值级记录 `applicable_scenarios`。

### 待确认

- `cross_scenario` 是否作为正式场景标识，还是改用空数组表示通用。
- 哪些 A1–A5 叶子只适用于自然语言交互；当前项目已确认本轮不处理图像类任务输入，但地图 / 视频指示仍存在于旧字典中。
- 场景 registry 与树节点启用条件的正式连接方式。

## 11. 决策 13：黑盒 SUT 只要求可声明 Trace，不收集思维链

暂定方案：

- A2、A3、A4 的内部过程主要通过结构化输入、工具 / 技能调用、公开状态变化、任务 Trace 和最终效果观察。
- 不要求、不保存也不评分模型的私有思维链。
- 若某过程字段对黑盒 SUT 不可观察，则不应直接作为强制评分项；可改为效果证据或从分母移除。
- 结构化 Trace schema 必须在评分前公开、版本化，并与 simulator 私有接口分离。

### 待确认

- SUT 必须公开的最小 Trace 字段集合。
- “不可观察项不进分母”的正式统计规则。
- 自然语言解释是否属于可选证据，如何防止“说对但没做”获得高分。

## 12. 决策 14：新 A5 L4 的偏好与高层通知只保留骨架

暂定方案：

- `jd-5.2` 先保存偏好类别、值表示、来源、适用范围、冲突状态和更新 / 撤销等结构。
- `jd-5.3` 先保存高层事件、通知对象、优先级、证据摘要、去重、升级与审计关联等结构。
- 正式 preference taxonomy、优先级、通知阈值、确认时限和 human burden 规则全部保持 TBD。
- 不将 A×L 的 L4 直接做成 JD 枚举值。

### 待确认

- preference taxonomy 的最终范围，以及哪些偏好允许影响执行。
- 偏好与法规、安全约束冲突时的固定优先级。
- 高层通知的必要性、去重、升级和人工负担如何定义与评分。
- 是否需要独立的“请求必要性 / 请求时机 / 请求质量”GT。

## 补充：A11 canonical 槽位、11.4/11.5 与白墙 v0.6 草案的处理

已采用保守方案：

- A11 的正式 `jd-11.1 控制质量先验`、`jd-11.2 控制工作机制`、`jd-11.3 控制处置策略适用集`逐字继承当前 66 槽位权威字典。
- `jd-11.1` 已按跟踪误差、瞬态与稳定性、控制与执行余量、巡检阶段质量 Profile 细分；具体误差和时间阈值保持 TBD。
- `jd-11.2` 已按控制目标与参考系、保持 / 跟踪模式、控制环与执行分配、控制器家族声明、状态与执行前提细分。PID、模型预测控制等只作为 SUT 声明 / Trace 候选，不作为默认 Seed 难度枚举。
- `jd-11.3` 已按异常触发、局部处置动作、策略适用条件、模式切换保护、处置后验证与跨能力升级细分；返航、降落和强制安全态继续引用 A16，不在 A11 重复定义。
- `PROPOSED-jd-11.4 控制执行条件与动态包络`用于补齐结构空档，细分平台与载荷执行基准、指令接口执行约束、运动与姿态动态包络、执行机构约束、包络修正与适用条件。它只引用 `jd-2.2` 平台静态档案，不重复保存平台参数；所有数值范围保持 TBD。
- 原 `jd-8.5` 的白墙窄缝内容不再直接充当通用 A11 树，而是先抽象为 `PROPOSED-jd-11.5 近障 / 受限空间控制场景 Profile`。该 Profile 描述障碍物几何、通行空间、布局、动态性、遭遇关系、派生控制压力，以及对 `jd-11.1/11.2/11.3` 和 `PROPOSED-jd-11.4` 的绑定。
- `PROPOSED-jd-11.5` 的 `canonical_slot` 保持空值，不计入当前 66 槽位。白墙窄缝原始草案作为其下的 `EXAMPLE-jd-11.5-white-wall-v0.6` 具体示例保留，原始候选取值可在节点详情中审阅。
- 平台静态参数引用 `jd-2.2`，航路和几何轨迹选择引用 A10，安全包络和兜底引用 A16；A11 保留闭环控制质量、控制机制、局部恢复和当前动态执行包络。

### 纹理、多通行空间与 `jd-0.3` 的本轮更正

- 表面纹理复杂度和目标 / 开口—背景局部对比度定义在 A6 `jd-6.2` 下；A11 `11.5.8` 只引用，不复制感知变量。
- `11.5.2` 使用“通行空间数量 + 实例集合”支持同一墙体或结构包含多个孔洞 / 缝隙；排列和间距只在数量不少于 2 时启用。
- `jd-0.1`–`jd-0.10` 全部继续作为跨 A 共享 global JD，不迁移为任何新 Axx；页面在同一个 `JD-global` 分支中完整展示。
- 静态光照基线与动态光线扰动分列，不能以 `光照=0` 表达“未启用扰动”。
- v0.6 中的纹理、对比度、缝数、排列和间距档位保留为团队资料候选，不视为跨场景正式 taxonomy 或阈值。

详细证据、原布尔字段解释和待确认项见 `docs/jd-variable-tree/JD业务变量树_version1_本轮更正审计.md`。

### 待确认

- `PROPOSED-jd-11.4` 是否进入正式 JD Catalog，正式名称是否采用“控制执行条件与动态包络”。
- `PROPOSED-jd-11.5` 是否进入正式 JD Catalog，及其正式编号。
- `jd-11.1` 的位置、姿态、速度、航向误差，超调、稳定时间、振荡和余量的正式单位、取值范围与判定方法。

## 15. 建议验收顺序

1. 先确认完整 A 域重编号和 local JD 前缀迁移。
2. 再确认 L0 数据合同。
3. 再确认四个侧 / 可见性字段的定义和允许值。
4. 逐个审阅 A1–A5 canonical 槽位下的子维度，只判断“保留 / 合并 / 删除 / TBD”。
5. 审阅 A4 / A5、A10 / A16 的责任边界。
6. 最后讨论候选值、阈值和 Pipeline 映射，避免在结构未定前固化接口。
