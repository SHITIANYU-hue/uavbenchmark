# 任务案例参考：城市高楼峡谷道路巡检

## 文件定位

本文件保留城市高楼峡谷违停车辆巡检的任务设计参考，不是当前 Stage 0 场景注册表中的预制答案，也不预设本次任务必然属于哪一个 A×L。

使用顺序为：

```text
固定业务场景与变量域
  → 人工描述本次任务
  → Config Agent 生成业务叙述、提取 JD 并提出 A×L 候选
  → 人工确认或修改能力等级
  → 形成最终 Task Template
```

当前 Stage 0 的正式机读入口是 `knowledge/business_scenario_registry.json`，其中包含高速、油气、桥梁、园区和其他巡检场景族。若未来需要把“城市高楼峡谷道路巡检”注册为独立业务场景，应另行经过业务确认后加入注册表。本文件依据 `【案例一】城市高楼峡谷违停车辆巡检.md` 的“0. 任务 Summary”和“1. 基本业务场景”整理。

## 0. 场景 Summary

业务场景固定为城市高楼峡谷道路巡检。无人机在指定道路巡检区域内运行，场景预先提供起飞点、巡检起点、巡检终点、固定道路航段、车辆及城市背景对象、连续 RGB 观测、多源定位观测和基础飞行状态。

道路中可包含违停车辆、正常停放车辆和相似干扰对象；目标可出现、离开、交错、短时遮挡或重新出现。高楼、玻璃幕墙、狭窄街谷和遮挡结构可形成 GNSS 多径、位置跳变、连续失锁或定位源不一致等定位退化条件。

固定航段推进、暂停推进、悬停或等效安全保持，以及速度、姿态、航向和位置的低层闭环控制，由外部 waypoint executor、仿真执行器或外部飞控提供。结构化外部决定接口可在收到异常事件后返回继续、等待恢复、返航或终止等后续动作。

Stage 0 只固定场景条件、可用观测、对象范围、外部设施和允许变量域。是否要求单次识别、持续维护、异常保护或自主处置，必须由 Stage 1 的人工任务描述决定。

## 1. 固定业务场景与变量

### 1.1 固定场景边界

- 城市高楼峡谷、狭窄街谷或高层建筑密集道路环境；
- 指定道路巡检区域、起飞点、巡检起终点和固定道路航段；
- 违停车辆、正常停放车辆及相似干扰对象；
- 连续 RGB、多源定位和基础飞行状态观测；
- 外部航段执行器、外部飞控和结构化外部决定接口；
- 可配置的定位退化事件域；
- 对 SUT 隐藏的真实车辆身份、违停标签、真实位姿和真实故障窗口。

### 1.2 业务变量绑定方式

| 变量 | 绑定方式 | Stage 0 的固定值或允许域 |
|---|---|---|
| `jd-0.1 任务时长` | 由任务提取 | 本次任务可包含正常阶段和异常阶段；具体时长未给出时保持 TBD |
| `jd-0.2 工作空间结构` | 固定值 | 城市高楼峡谷、狭窄街谷或高层建筑密集道路环境 |
| `jd-0.3 扰动谱` | 固定取值域 | GNSS 多径、位置跳变、连续失锁、定位源不一致；是否启用由任务决定 |
| `jd-0.4 在场人角色` | 固定值 | 外部操作员或地面站任务监督者 |
| `jd-0.5 人机确认协议` | 固定值 | 结构化异常事件和外部决定接口；不评价自然语言协认 |
| `jd-0.6 合规域` | 固定取值域 | 道路巡检边界、禁飞区和基础运行限制；具体几何待实例化 |
| `jd-0.7 任务完成判据` | 由任务提取 | 根据本次人工任务责任生成；未知阈值和数量保持 TBD |
| `jd-0.9 载荷集` | 固定值 | RGB 相机 |
| `jd-0.10 安全包络先验` | 固定取值域 | 速度、净空、禁飞区、姿态和资源约束；具体数值保持 TBD |
| `jd-5.1 业务对象类目` | 固定值 | 违停车辆、正常停放车辆及相似干扰对象 |
| `jd-5.2 典型场景特征` | 固定取值域 | 多目标、遮挡、交错、离开、重新出现、尺度和视角变化 |
| `jd-5.3 感知模态集` | 固定值 | 连续多帧 RGB 观测 |
| `jd-6a.1 定位源需求` | 固定取值域 | GNSS/RTK、VIO、IMU、磁力计、气压计或其他允许源 |
| `jd-6a.2 位姿质量先验` | 固定取值域 | 位置、航向、协方差、残差、质量等级或等价可信性表达 |
| `jd-7.1 航路拓扑` | 固定取值域 | 起飞点、巡检起终点、固定航点、道路航段和禁飞约束 |
| `jd-7.2 导航工作机制` | 固定边界 | 外部执行器推进固定航段；SUT 的导航责任由任务决定 |
| `jd-8.1 控制质量先验` | 固定取值域 | 速度、姿态、航向、净空和保持误差；允许数值保持 TBD |
| `jd-8.2 控制工作机制` | 固定边界 | 外部飞控完成闭环执行；SUT 的状态理解或建议责任由任务决定 |

### 1.3 外部运行依赖

| Dependency | 固定责任 | 是否计入 SUT 能力 |
|---|---|---|
| `external_waypoint_executor` | 推进固定道路航段，落实暂停推进、悬停或等效安全保持 | 否 |
| `external_flight_controller` | 完成速度、姿态、航向和位置的低层闭环控制 | 否 |
| `external_decision_port` | 接收结构化异常事件，并按任务约定返回后续动作 | 否 |

## 2. Stage 1 人工任务输入

人工输入只回答以下问题：

1. 希望无人机在该固定场景中完成什么；
2. 结果是一次输出，还是需要持续维护和更新；
3. 出现异常时系统需要识别、保护、上报，还是自主处置；
4. 哪些动作由外部执行器或人完成；
5. 期望输出什么，以及任务何时结束。

人工输入不需要填写 A×L 或 JD 编号。例如：

> 请让无人机沿固定道路航段持续巡检违停车辆。车辆出现、短时遮挡或重新出现时，系统应保持同一目标身份，合并重复观测并持续更新巡检报告。定位观测明显退化时，系统应标记位姿不可信，请求外部执行器进入安全保持，并通过结构化接口上报异常后等待外部决定。系统只输出航路风险和飞行状态建议，不自主修改航路，也不执行低层飞行控制。

## 3. Agent 输出与人工复核

Config Agent 输出三个相互分离的候选产物：

- 面向业务人员的自然语言任务叙述，正文不显示 A×L、GAL 或 JD 内部编号；
- 结构化 JD 提取结果，区分固定场景值、任务原文值与 TBD；
- 结构化 A×L 候选及 Runtime Dependencies。

能力等级不是 Agent 自动批准的结论。Stage 2 必须提供 L1、L2、L3、L4 人工选项。例如 Agent 提出某能力为 L2，但人工确认任务已经包含异常保护和上报责任时，可以将其修正为 L3。修改只影响本次任务候选，并应记录人工确认来源。

## 4. 不得自动补全

- 具体观测时长；
- 车辆数量和业务通过阈值；
- 定位可信阈值与退化判据数值；
- 安全包络数值；
- 外部决定接口 schema、超时和枚举；
- simulator 或真实飞控接口。

上述内容没有人工或受审配置来源时必须保持 TBD。
