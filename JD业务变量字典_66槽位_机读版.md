# UAV Benchmark JD 业务变量字典

> **版本：** `jd-66-slots-ability-id-v3-2026-07-20`
> **范围：** 10 个 JD-global + 56 个 JD-local = 66 个槽位；覆盖 17 个 A 域。
> **用途：** Config Agent、Task Template、实例生成、检索、校验和代码侧 schema 映射。

## 0. 版本与使用说明

- 本文件保留原 PNG 中 66 个 JD 槽位的业务语义，2026-07-19 曾采用团队确认的 `ability-id-v2` 编号，2026-07-20 起采用连续化的 `ability-id-v3` 编号。
- `LGA_含业务变量.png` 仍显示旧编号，只能作为语义来源，不能再作为当前 ID 的机器读取来源。
- 截图中个别说明以省略号结尾；本文件对其做了**保守的语义归一化**，不是逐字 OCR 转录。
- 本图的 A1 只有 `jd-1.1`–`jd-1.3`。部分旧材料曾出现 `jd-1.4`，但它**不属于本图的 66 槽位版本**，代码中不要无版本判断地混用。
- `owner_a` 只用于 local 变量；global 变量通过 `related_a` 表示被哪些 A 引用。
- 关系矩阵中的“引用/拥有”不等同于某个任务一定显式测到该 A。

2026-07-19 的旧→新（v1→v2）能力编号为：`A14→A5`、`A5→A6`、`A6a/A6b→A7a/A7b`、`A11→A8`、`A7→A9`、`A8→A10`、`A9a/A9b→A11a/A11b`、`A10→A12`、`A12→A14`；A1–A4、A13、A15 不变。local JD 的数字前缀同步迁移。此为历史记录，保留不改。

2026-07-22 的当前连续编号为：`A7a→A7`、`A7b→A8`、`A8→A9`、`A9→A10`、`A10→A11`、`A11a→A12`、`A11b→A13`、`A12→A14`、`A15→A15`、`A13→A16`、`A14→A17`；A1–A6 不变。最终尾部能力固定为 `A15=审计日志`、`A16=合规申报`、`A17=安全包络`，local JD 数字前缀同步迁移。当前机读编号以本文件为准。

## 1. 建议的代码检索字段

| 字段 | 含义 | 示例 |
|---|---|---|
| `id` | 稳定主键 | `jd-7.3` |
| `scope` | `global` 或 `local` | `local` |
| `g_group` | local 所属能力组 | `G2` |
| `owner_a` | local 所属 A；global 为 `null` | `A7` |
| `ability` | A 域中文名称 | `自定位 / 姿态航向估计` |
| `name` | JD 变量名称 | `定位策略适用集` |
| `description` | 变量语义说明 | — |
| `related_a` | 引用或拥有该变量的 A 列表 | `[A7]` |

## 6. 检索与版本约束

1. 优先按 `id` 精确检索，不要只按中文名称匹配。
2. 解析 YAML 时，以 `variables` 为主表，以 `reverse_index_by_a` 为反向索引。
3. global 变量没有唯一 owner；不要把 `related_a` 的第一个元素当成 owner。
4. local 变量的 `related_a` 当前只含所属 A；跨能力复用应通过 global 变量或任务级显式映射表达。
5. Config 或 episode 应保存 `schema_version`，防止与含 `jd-1.4` 等旧版本混用。
